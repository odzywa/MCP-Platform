MCP Platform v1.3

Dokumentacja platformy MCP

Platforma control plane + generic runtime do tworzenia, wdrażania i zarządzania izolowanymi serwerami MCP (Model Context Protocol). Bez pisania kodu — konfigurujesz narzędzia w UI, platforma generuje artefakty i deployuje kontenery Docker lub pody Kubernetes.

Aktualizacja: lipiec 2026  ·  Docker Compose + Kubernetes/OpenShift + System zatwierdzeń

Architektura

Platforma składa się z trzech warstw: control plane (UI + API + DB), operator (reconciler) i kontenery runtime (serwery MCP).

Zasada działania Control plane nigdy nie uruchamia kontenerów — zapisuje deployment_requests do SQLite. Operator pobiera żądania i wykonuje je przez Docker SDK lub Kubernetes API.

Instalacja: Docker Compose

  1. Klonuj repozytorium 
    git clone https://github.com/MCP-Github/MCPPlatform
cd MCPPlatform
  2. Zbuduj obrazy runtime 
    docker compose --profile build-only build
  3. Uruchom platformę 
    docker compose up -d mcp-platform mcp-platform-operator
  4. Otwórz UI

    UI dostępne pod http://localhost:18100 (lub http://mcp.dom jeśli skonfigurowany DNS).

    Pierwsze logowanie: admin / admin — zmień hasło po pierwszym logowaniu.

Instalacja: Kubernetes / OpenShift

Katalog MCCPLATFORMforKUBERNETES/ zawiera kompletny stack K8s/OCP: manifesty YAML, skrypt deploy.sh i osobny operator z KubernetesDeploymentDriver.

Wymagania
OpenShift 4.x lub Kubernetes 1.26+
LVM Storage Operator (lub inny StorageClass)
Image Registry (wewnętrzny route OCP lub Harbor)
oc / kubectl + uprawnienia admin
Struktura plików
config.env — konfiguracja
deploy.sh — skrypt deploymentu
k8s/ — manifesty YAML
operator/ — operator K8s

1. Konfiguracja config.env

MCCPLATFORMforKUBERNETES/config.env
# Zewnętrzny route do pushowania obrazów (podman push)
REGISTRY=default-route-openshift-image-registry.apps.twoj-klaster.dom/mcp-platform

# Wewnętrzny svc — pod nim pody pobierają obrazy (imagePullPolicy: Always)
PULL_REGISTRY=image-registry.openshift-image-registry.svc:5000/mcp-platform

# Domena aplikacji OCP
APPS_DOMAIN=apps.twoj-klaster.dom

# Namespace Kubernetes
NAMESPACE=mcp-platform

# StorageClass do PVC (lvms-storage dla LVM Storage Operator)
STORAGE_CLASS=lvms-storage
Dwa rejestry REGISTRY to zewnętrzny route — używany przez podman push. PULL_REGISTRY to wewnętrzny svc — używany przez pody w manifeście. Nigdy nie mieszaj ich ról.

2. Uruchomienie deploy.sh

cd MCCPLATFORMforKUBERNETES
chmod +x deploy.sh
./deploy.sh

Skrypt automatycznie:

  • Loguje się do rejestru OCP przez oc registry login
  • Buduje obrazy: mcp-platform-control-plane, mcp-platform-operator-k8s, mcp-runtime-shell, mcp-runtime-http-gateway, mcp-runtime-openapi
  • Pushuje je przez podman push --tls-verify=false (TLS terminowany przez Route)
  • Aplikuje manifesty YAML przez oc apply -k k8s/

3. Weryfikacja

oc -n mcp-platform get pods
oc -n mcp-platform get routes

UI dostępne pod https://mcp-platform-mcp-platform.<APPS_DOMAIN>.

4. OpenShift MCP — auto-konfiguracja (opcjonalna)

Platforma automatycznie tworzy runtime openshift-monitor w statusie draft przy starcie. Żeby go wdrożyć razem z deploy.sh, wypełnij dwa pola w config.env przed uruchomieniem skryptu:

Utwórz dedykowany Service Account Nie używaj tokena admina — utwórz osobne SA z minimalnym zakresem uprawnień.
Krok 1 — SA + token (jednorazowo)
oc create sa mcp-admin -n mcp-platform
oc adm policy add-cluster-role-to-user cluster-admin -z mcp-admin -n mcp-platform

# Token ważny 1 rok:
OC_MCP_TOKEN=$(oc create token mcp-admin -n mcp-platform --duration=8760h)
OC_MCP_SERVER=$(oc whoami --show-server)

echo "OC_MCP_TOKEN=$OC_MCP_TOKEN" >> config.env
echo "OC_MCP_SERVER=$OC_MCP_SERVER" >> config.env
Krok 2 — deploy
./deploy.sh
# Krok 6 automatycznie: loguje się do API platformy, ustawia credentials, deployuje openshift-monitor
Wynik Runtime openshift-monitor jest uruchomiony ~30s po zakończeniu deploy.sh. UI → openshift-monitor → Auth → Generate token → wklej jako X-API-Key w kliencie AI.

2Secrets → + Add → OC_TOKEN = token SA, OC_SERVER = URL API

Rejestr obrazów OpenShift

Domyślnie OpenShift nie udostępnia zewnętrznego route do rejestru obrazów. Aktywacja:

  1. Utwórz PVC dla rejestru 
    oc -n openshift-image-registry patch configs.imageregistry.operator.openshift.io/cluster \
  --type merge -p '{"spec":{"storage":{"pvc":{"claim":""}}}}'
  2. Włącz zewnętrzny route 
    oc -n openshift-image-registry patch configs.imageregistry.operator.openshift.io/cluster \
  --type merge -p '{"spec":{"defaultRoute":true}}'
  3. Zmień strategię rollout na Recreate (wymagane dla RWO PVC) 
    oc -n openshift-image-registry patch configs.imageregistry.operator.openshift.io/cluster \
  --type merge -p '{"spec":{"rolloutStrategy":"Recreate"}}'
  4. Zaloguj się do rejestru 
    oc registry login --skip-check
    UwagaNie używaj docker login -p $(oc whoami -t) — token ma dłuższą strukturę niż Docker acceptuje. Użyj oc registry login.

Certyfikat CA OpenShift

OCP używa samopodpisanego CA. Pobierz i zainstaluj go w systemie — wymagane aby curl, podman i inne narzędzia ufały Route TLS.

  1. Pobierz certyfikat CA 
    oc extract configmap/kube-root-ca.crt -n openshift-config --to=- > ocp-ca.crt
  2. Zainstaluj w systemie (RHEL/Fedora/CentOS) 
    sudo cp ocp-ca.crt /etc/pki/ca-trust/source/anchors/
sudo update-ca-trust
  3. Zainstaluj w systemie (Debian/Ubuntu) 
    sudo cp ocp-ca.crt /usr/local/share/ca-certificates/ocp-ca.crt
sudo update-ca-certificates
Weryfikacja curl https://mcp-platform-mcp-platform.apps.twoj-klaster.dom/health powinno zwrócić {"status":"ok"} bez błędu TLS.

Bezpieczeństwo: runtime Shell

Runtime shell wykonuje komendy systemowe zgodnie z konfiguracją narzędzi. Zaimplementowany model eliminuje klasyczne wektory ataku przez wstrzyknięcie komend.

Model bez shell=True

shell=True usunięte Cały kod wykonuje podprocesy przez subprocess.run(shell=False) lub subprocess.Popen(shell=False). Metacharakter |, ;, && w danych użytkownika to zwykłe argv tokens — nie operatory shella.

Jak działają potoki (pipes)

Potoki są modelowane jawnie na poziomie szablonu narzędzia — separator | musi pojawić się jako osobny token w definicji komendy (nie w danych użytkownika):

✅ Bezpieczny pipeline
{"command": ["printf","%s","${yaml}","|","oc","apply","-f","-"]}

| pochodzi z szablonu → Popen chain

✅ Inertny token od użytkownika
// użytkownik podaje: "foo | rm -rf /"
${*args} → argv: ["foo","|","rm","-rf","/"]
// "|" to string argument, nie operator

Rozwinięcia zmiennych

SkładniaZachowanieAtak
${var}JEDEN element argv (spacje nie dzielą)blokowany
${*var}shlex.split() → extend argv (bez rejoin)inertny
| w szablonieSeparator pipeline → Popen chainby design
| od użytkownikaZwykły token w argvblokowany

Bezpieczeństwo: polityka narzędzi

Każdy runtime ma plik policy.json kontrolujący co wolno wykonać. Politykę ustawiasz w UI → serwer → Bezpieczeństwo.

Przykład policy.json — OpenShift read-only
{
  "allowed_binaries": ["oc", "kubectl", "jq"],
  "blocked_commands": ["delete", "apply", "patch", "exec"],
  "require_read_only": true,
  "block_write_tools": true,
  "block_destructive_tools": true,
  "timeout_seconds": 30,
  "max_response_bytes": 5242880,
  "require_approval_for": []
}
Przykład policy.json — OpenShift z zatwierdzeniami
{
  "allowed_binaries": ["oc", "kubectl", "jq", "printf"],
  "blocked_commands": [],
  "require_read_only": false,
  "block_write_tools": false,
  "require_approval_for": "auto",
  "approval_timeout_seconds": 300
}

Allowlist binaries

Pole allowed_binaries sprawdzane jest dla każdego etapu pipeline niezależnie. Jeśli lista jest ustawiona, binarka nieznajdująca się na liście jest blokowana nawet jeśli pierwszy etap przeszedł weryfikację.

Bezpieczeństwo: SSRF i sieć

Ochrona przed SSRF

Control plane sprawdza każdy zewnętrzny URL przed wysłaniem żądania HTTP. Funkcja _is_safe_fetch_url():

  • Blokuje schematy inne niż http/https
  • Blokuje znane lokalne nazwy: localhost, 169.254.169.254, metadata.google.internal
  • Blokuje numeryczne adresy prywatne (RFC 1918, loopback, link-local)
  • Dla nazw domenowych: rozwiązuje DNS przez socket.getaddrinfo() i sprawdza każdy zwrócony adres IP — blokuje DNS rebinding
Dotyczy Import paczek narzędzi z URL, rejestracja zewnętrznych serwerów MCP, probing endpointów MCP.

Izolacja sieciowa runtime

Kontenery runtime działają w sieci Docker ai-net (izolowanej od hosta). Control plane i operator komunikują się przez tę sieć. Zewnętrzne porty (19000–19800) wystawiane są opcjonalnie przez MCP_RUNTIME_PUBLIC_PORT_BASE.

Cookie i sesje

Sesja użytkownika przechowywana w cookie mcp_session:

  • httponly=True — niedostępne z JavaScript
  • samesite="lax" — ochrona przed CSRF dla nawigacji cross-site
  • secure=True — włącz przez MCP_HTTPS_ONLY=1 gdy platforma za HTTPS termination (np. OpenShift Route)

System zatwierdzeń (Human-in-the-Loop) NOWE

Narzędzia z trybem write lub destructive mogą wymagać potwierdzenia użytkownika przed wykonaniem. Zatwierdzenie odbywa się bezpośrednio w chacie AI — nie potrzeba osobnego panelu ani dodatkowych akcji w UI.

Jak to działa

AI wywołuje narzędzie
→
Runtime sprawdza policy
→
⚠️ approval_required
→
AI pyta w chacie
→
Użytkownik: "tak"
→
✅ AI wywołuje ponownie → execute

Runtime zwraca do AI komunikat z opisem komendy i instrukcją. AI pyta użytkownika w chacie. Po potwierdzeniu AI wywołuje to samo narzędzie ponownie — runtime automatycznie wykonuje komendę.

Dwa sposoby potwierdzenia

SposóbJak działaKompatybilność
Auto-confirm przy ponownym wywołaniu Pierwsze wywołanie rejestruje pending. Drugie identyczne wywołanie (ten sam tool + te same args) w oknie approval_timeout_seconds jest automatycznie zatwierdzone — bez żadnych dodatkowych parametrów. Wszystkie modele AI
__confirm parametr AI dodaje __confirm="yes" (lub "tak", "true", "1", "ok" itd.) do parametrów wywołania. Parametr jest zdefiniowany w schemacie każdego narzędzia. Modele które potrafią dodać nowy parametr do istniejącego wywołania
Dlaczego auto-confirm działa z każdym modelem? Niektóre modele (np. lokalne jak Qwen, Llama) nie potrafią dodać parametru __confirm do ponownego wywołania narzędzia. Auto-confirm rozwiązuje ten problem — AI po prostu wywołuje to samo narzędzie z tymi samymi argumentami, co jest naturalnym zachowaniem po potwierdzeniu przez użytkownika.

Konfiguracja

Wartość require_approval_forZachowanie
nie ustawione lub ""Brak zatwierdzeń — domyślne
"auto"Auto-detekcja: mode=write/destructive LUB nazwa narzędzia zawiera słowo kluczowe (delete, create, apply, patch, scale…)
["destructive"]Tylko narzędzia z mode=destructive (delete)
["write", "destructive"]Narzędzia write (create/apply) i destructive (delete)
Zalecana konfiguracja — auto z prefiksami komend
{
  "require_approval_for": "auto",
  "require_approval_for_prefixes": ["oc delete", "oc apply", "kubectl delete"],
  "approval_timeout_seconds": 300
}
Nie mieszaj z blocked_command_prefixes Jeśli chcesz wymagać zatwierdzenia dla danego prefiksu — użyj require_approval_for_prefixes. Nie dodawaj tych samych prefiksów do blocked_command_prefixes, bo zatwierdzenie nie pomoże — po potwierdzeniu runtime pomija blokadę prefiksów, ale nadal musisz mieć komendę na liście dozwolonych binariów (allowed_binaries).

Uwierzytelnianie MCP NOWE

Każdy serwer MCP może być zabezpieczony tokenem Bearer, który musi być podany przez klienta AI (Claude Desktop, Cline, Continue itp.). Bez tokenu serwer pozostaje otwarty — funkcja jest opcjonalna i per-runtime.

Jak to działa

Klient AI
→
Authorization: Bearer <token>
→
Middleware runtime
→
✅ Autoryzowany → narzędzie

Brak lub błędny token → 401 Unauthorized z nagłówkiem WWW-Authenticate: Bearer.

Włączanie tokenu (UI)

  1. Wejdź na stronę szczegółów runtime'u (Runtimes → nazwa)
  2. Przejdź do zakładki 🔐 Auth
  3. Kliknij + Generuj token
  4. Skopiuj token i wklej do konfiguracji klienta AI

Token jest generowany przez secrets.token_urlsafe(32) — 256 bitów entropii. Rotacja: kliknij ↻ Regeneruj token — stary token natychmiast przestaje działać.

Zero downtime Zmiana tokenu nie wymaga restartu kontenera. Token jest zapisywany do runtime-config.json i ładowany przez /reload automatycznie po wygenerowaniu.

Nagłówki — obsługiwane formaty

NagłówekFormatOpis
AuthorizationBearer <token>Standard OAuth2 / RFC 6750 — zalecany
X-API-Key<token>Prostszy format — sam token, bez prefiksu Bearer
Zmienne środowiskowe NIE zastępują nagłówka Niektóre klienty (np. OpenCode) mają sekcję "Zmienne środowiskowe" w konfiguracji MCP. Dla serwerów zdalnych (transport zdalny-SSE lub streamable-HTTP) te zmienne nie są wysyłane do serwera — służą tylko do lokalnych procesów (stdio). Token zawsze musi iść w nagłówku żądania.

Ścieżki wyłączone z autoryzacji

Następujące endpointy są zawsze publiczne (nie wymagają tokenu):

  • /health — sprawdzanie stanu kontenera przez operator
  • /reload — przeładowanie konfiguracji przez operator (tylko runtime-shell i http-gateway)

Bezpieczeństwo porównania tokenów

Porównanie odbywa się przez secrets.compare_digest() — odporne na timing attacks. Token nigdy nie jest logowany w żadnym pliku ani bazie danych oprócz runtimes.mcp_auth_token (plaintext — nie używaj tego na produkcji bez szyfrowania bazy).

Uwaga: plaintext w SQLite Kolumna runtimes.mcp_auth_token przechowuje token w formie jawnej. W środowisku produkcyjnym zalecane jest szyfrowanie bazy lub użycie zewnętrznego vault. Jest to znane ograniczenie opisane w dokumentacji architektury.

allowed_command_prefixesstring[]Allowlist — tylko komendy zaczynające się od tych prefiksów są dozwolone. Jeśli pusta, wszystkie przechodzą.
blocked_command_prefixesstring[]Blokuje komendy o podanych prefiksach. Pomijane gdy __confirm lub auto-confirm z okna.

ModeZastosowanieApproval autoPrzykłady

Tryb auto W trybie "require_approval_for": "auto" wykrywanie jest dwuetapowe: najpierw sprawdzany jest mode narzędzia, następnie słowa kluczowe w nazwie. Jeśli admin błędnie oznaczy narzędzie jako read-only ale w nazwie jest delete — i tak zostanie wymagane zatwierdzenie.

Cykl życia runtimeu — deploy, konfiguracja, reload

Każdy serwer MCP przechodzi przez kilka etapów. Zrozumienie co wymaga Reload, a co Redeploy pozwala unikać niepotrzebnych restartów.

Pierwsze uruchomienie

  1. Utwórz runtime — UI → Szybki start (lub Kreator zaawansowany) → wybierz klasę (np. shell-readwrite) → Utwórz serwer.
  2. Dodaj narzędzia — zakładka Narzędzia → + Dodaj narzędzie → wypełnij nazwę, komendę, mode, schemat argumentów.
  3. Ustaw politykę — zakładka Polityka → ustaw allowed_binaries, require_approval_for itp. → Zapisz.
  4. Dodaj sekrety/zmienne — zakładka Sekrety → + Dodaj (wartości wstrzykiwane jako env vars do kontenera).
  5. Wdróż — przycisk ▶ Deploy (lub operator automatycznie wykrywa nowy rekord). Status zmieni się na running.
  6. Podłącz klienta AI — skopiuj endpoint MCP z nagłówka strony (np. http://localhost:19001/mcp) do konfiguracji Claude Desktop / Continue / Open WebUI.

Co wymaga tylko Reload (bez restartu kontenera)

Reload (POST /reload na runtimeie) przeładowuje pliki konfiguracyjne z dysku bez ubijania procesu. Jest automatycznie wyzwalany przez control plane po zapisie zmian.

ZmianaJak zastosowaćReload automatyczny?
Edycja narzędzia (nazwa, komenda, schemat, mode)UI → Narzędzia → Edytuj → Zapisz✅ Tak — control plane woła /reload
Zmiana polityki (allowed_binaries, approval, timeouty)UI → Polityka → Zapisz politykę shell✅ Tak
Zmiana tokenu authUI → Auth → Generuj / Regeneruj token✅ Tak
Import paczki narzędziUI → Import JSON✅ Tak
Reload nie przerywa działających wywołań Proces runtimeu działa dalej — tylko config files są ponownie wczytywane. Klienci MCP nie tracą połączenia.

Co wymaga Redeploy (restart kontenera)

ZmianaJak zastosować
Dodanie / usunięcie sekretów (env vars)UI → Sekrety → zapisz → kliknij 🔄 Redeploy
Zmiana klasy runtimeu (inny obraz Docker)UI → Redeploy
Zmiana limitu pamięci / CPU (wymaga modyfikacji polityki)UI → Redeploy
Nowa wersja obrazu runtime-shell po ./deploy.shUI → Przebuduj obraz + Deploy (na K8s: robi rollout restart)
Sekrety wymagają Redeploy Zmienne środowiskowe są wstrzykiwane do kontenera przy starcie. Zmiana wartości sekretu nie jest widoczna dla działającego kontenera dopóki nie zostanie zrobiony Redeploy.

Jak Reload działa technicznie

Control plane zapisuje artefakty do ./data/configs/<runtime-id>/ (Docker) lub ConfigMap (K8s), następnie woła POST /reload na endpoincie runtimeu. Runtime wczytuje ponownie:

  • tools.json — definicje narzędzi z komendami
  • policy.json — polityka bezpieczeństwa
  • runtime-config.json — metadata + token auth
  • adapter-config.json — konfiguracja adapterów

runtime-env.json (env vars z sekretów) nie jest przeładowywany — wymaga restartu procesu.

Ręczny reload (jeśli automatyczny nie zadziałał)

# Docker — bezpośrednio na kontenerze
curl -X POST http://localhost:19001/reload

# OpenShift — przez route runtimeu
curl -X POST https://mcp-runtime-<id>-mcp-platform.apps.<domain>/reload

Rebuild obrazu runtime

Docker Compose

  1. Przebuduj obraz

Kubernetes / OpenShift

Na K8s nie istnieje BuildConfig — obrazy buduje deploy.sh na zewnątrz. Akcja Przebuduj obraz + Deploy pomija budowanie i robi tylko rollout restart z imagePullPolicy: Always.

  1. Przebuduj i pushuj obraz
    cd MCCPLATFORMforKUBERNETES
./deploy.sh
    lub ręcznie: 
    docker build -t $REGISTRY/mcp-runtime-shell:latest ../runtime-shell
podman push --tls-verify=false $REGISTRY/mcp-runtime-shell:latest
  2. W UI — Runtimes → serwer → Przebuduj obraz + Deploy (wyzwala rollout restart → pod ściąga nowy obraz)

Rozwiązywanie problemów

Ogólne

ProblemRozwiązanie
Pod runtime w stanie ImagePullBackOffSprawdź czy MCP_RUNTIME_IMAGE_REGISTRY_PREFIX wskazuje na wewnętrzny svc (nie zewnętrzny route). Pod nie może używać zewnętrznego route do pull.
403 Forbidden przy Przebuduj obraz + Deploy na K8sNa K8s nie ma BuildConfig — akcja poprawnie pomija build i robi tylko redeploy. Jeśli 403, sprawdź uprawnienia ServiceAccount operatora.
Runtime w stanie failed: SCC forbidden runAsUserUsuń runAsUser: 1000 i fsGroup: 1000 z pod securityContext. Zostaw tylko runAsNonRoot: true.
oc: command not found w deploy.shPobierz oc do katalogu: wget https://...tar.gz && tar xvf oc.tar. deploy.sh automatycznie dodaje $PWD do PATH.
LVMCluster stuck ProgressingPo restarcie klastra: sudo rm -f /var/lock/vgmanager/vgmanager.lock na węźle, usuń pod vg-managera.
LVMCluster Failed: already exists in filesystem(a) Nazwa VG pokrywa się z nazwą urządzenia — zmień name na inne niż vdc. (b) Stare sygnatury PV na dysku — pvremove -ff /dev/vdc.
docker push błąd TLS do rejestru OCPUżyj podman push --tls-verify=false zamiast docker push. Docker wymaga restartu po dodaniu insecure-registries.
Zatwierdzenie nie działa — AI wywołuje ponownie ale nadal blokowaneSprawdź czy to to samo narzędzie z tymi samymi argumentami. Auto-confirm działa tylko gdy hash (tool_name + args) jest identyczny. Restartowanie poda kasuje pending approvals.
Zwiększ okno zatwierdzeniaUstaw approval_timeout_seconds w policy JSON (domyślnie 300s). Po restarcie poda pending approvals są kasowane.
