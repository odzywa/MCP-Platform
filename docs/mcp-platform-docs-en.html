MCP Platform Documentation

A control plane + generic runtime system for creating, deploying, and managing isolated MCP (Model Context Protocol) servers. No code to write — configure tools in the UI, the platform generates artifacts and deploys Docker containers or Kubernetes pods.

Updated: July 2026  ·  Docker Compose + Kubernetes/OpenShift + Human-in-the-Loop Approvals

Architecture

The platform consists of three layers: a control plane (UI + API + DB), an operator (reconciler), and runtime containers (MCP servers).

Installation: Kubernetes / OpenShift

The MCCPLATFORMforKUBERNETES/ directory contains a complete K8s/OCP stack: YAML manifests, deploy.sh script, and a separate operator with KubernetesDeploymentDriver.

Requirements
OpenShift 4.x or Kubernetes 1.26+
LVM Storage Operator (or other StorageClass)
Image Registry (OCP internal route or Harbor)
oc / kubectl + admin permissions
File structure
config.env — configuration
deploy.sh — deployment script
k8s/ — YAML manifests
operator/ — K8s operator

1. Fill in config.env

MCCPLATFORMforKUBERNETES/config.env
# External route for pushing images (podman push)
REGISTRY=default-route-openshift-image-registry.apps.your-cluster.dom/mcp-platform

# Internal svc — used by pods to pull images (imagePullPolicy: Always)
PULL_REGISTRY=image-registry.openshift-image-registry.svc:5000/mcp-platform

# OCP apps domain
APPS_DOMAIN=apps.your-cluster.dom

# Kubernetes namespace
NAMESPACE=mcp-platform

# StorageClass for PVC (lvms-storage for LVM Storage Operator)
STORAGE_CLASS=lvms-storage
Two registries REGISTRY is the external route — used by podman push. PULL_REGISTRY is the internal svc — used by pods in the manifest. Never mix their roles.

2. Run deploy.sh

cd MCCPLATFORMforKUBERNETES
chmod +x deploy.sh
./deploy.sh

The script automatically:

  • Logs in to the OCP registry via oc registry login
  • Builds images: mcp-platform-control-plane, mcp-platform-operator-k8s, mcp-runtime-shell, mcp-runtime-http-gateway, mcp-runtime-openapi
  • Pushes them via podman push --tls-verify=false (TLS terminated by Route)
  • Applies YAML manifests via oc apply -k k8s/

3. Verify

oc -n mcp-platform get pods
oc -n mcp-platform get routes

UI available at https://mcp-platform-mcp-platform.<APPS_DOMAIN>.

4. OpenShift MCP — auto-setup (optional)

The platform automatically seeds the openshift-monitor runtime in draft status on startup. To deploy it together with deploy.sh, fill in two fields in config.env before running the script:

Use a dedicated Service Account Don't use an admin token — create a separate SA with a minimal permission scope.
Step 1 — SA + token (once)
oc create sa mcp-admin -n mcp-platform
oc adm policy add-cluster-role-to-user cluster-admin -z mcp-admin -n mcp-platform

# Token valid for 1 year:
OC_MCP_TOKEN=$(oc create token mcp-admin -n mcp-platform --duration=8760h)
OC_MCP_SERVER=$(oc whoami --show-server)

echo "OC_MCP_TOKEN=$OC_MCP_TOKEN" >> config.env
echo "OC_MCP_SERVER=$OC_MCP_SERVER" >> config.env
Step 2 — deploy
./deploy.sh
# Step 6 automatically: logs in to platform API, sets credentials, deploys openshift-monitor
Result The openshift-monitor runtime is running ~30s after deploy.sh completes. UI → openshift-monitor → Auth → Generate token → paste as X-API-Key in your AI client.

OpenShift Image Registry

By default OpenShift does not expose an external route to the image registry. To enable it:

  1. Create PVC for the registry 
    oc -n openshift-image-registry patch configs.imageregistry.operator.openshift.io/cluster \
  --type merge -p '{"spec":{"storage":{"pvc":{"claim":""}}}}'
  2. Enable external route 
    oc -n openshift-image-registry patch configs.imageregistry.operator.openshift.io/cluster \
  --type merge -p '{"spec":{"defaultRoute":true}}'
  3. Switch rollout strategy to Recreate (required for RWO PVC) 
    oc -n openshift-image-registry patch configs.imageregistry.operator.openshift.io/cluster \
  --type merge -p '{"spec":{"rolloutStrategy":"Recreate"}}'
  4. Log in to the registry 
    oc registry login --skip-check
    NoteDo not use docker login -p $(oc whoami -t) — the token format is longer than Docker accepts. Use oc registry login instead.

LVM Storage Operator (SNO)

For Single Node OpenShift clusters with LVM Storage:

LVMCluster — important note on VG naming
apiVersion: lvm.topolvm.io/v1alpha1
kind: LVMCluster
metadata:
  name: mcp-lvmcluster
  namespace: openshift-storage
spec:
  storage:
    deviceClasses:
      - name: storage          # ← must NOT match the device name (/dev/vdc → name: storage, NOT: vdc)
        thinPoolConfig:
          name: thin-pool-1
          sizePercent: 90
          overprovisionFactor: 10
        deviceSelector:
          paths:
            - /dev/vdc
If VG is locked After cluster restart the vg-manager may be locked. Remove the lock file: sudo rm -f /var/lock/vgmanager/vgmanager.lock on the node, then delete the vg-manager pod.

OpenShift CA Certificate

OCP uses a self-signed CA. Download and install it on your system — required so that curl, podman, and other tools trust Route TLS.

  1. Download the CA certificate 
    oc extract configmap/kube-root-ca.crt -n openshift-config --to=- > ocp-ca.crt
  2. Install system-wide (RHEL/Fedora/CentOS) 
    sudo cp ocp-ca.crt /etc/pki/ca-trust/source/anchors/
sudo update-ca-trust
  3. Install system-wide (Debian/Ubuntu) 
    sudo cp ocp-ca.crt /usr/local/share/ca-certificates/ocp-ca.crt
sudo update-ca-certificates
Verify curl https://mcp-platform-mcp-platform.apps.your-cluster.dom/health should return {"status":"ok"} without a TLS error.

Security: Shell Runtime

The shell runtime executes system commands as configured in tool definitions. The implemented model eliminates classic command injection attack vectors.

No shell=True

shell=True removed All subprocesses run via subprocess.run(shell=False) or subprocess.Popen(shell=False). Metacharacters |, ;, && in user input are plain argv tokens — not shell operators.

How pipelines work

Pipelines are modelled explicitly at the tool template level — the | separator must appear as a separate token in the command definition (not in user data):

✅ Safe pipeline
{"command": ["printf","%s","${yaml}","|","oc","apply","-f","-"]}

| comes from the template → Popen chain

✅ Inert token from user
// user passes: "foo | rm -rf /"
${*args} → argv: ["foo","|","rm","-rf","/"]
// "|" is a string argument, not an operator

Variable expansion

SyntaxBehaviourInjection
${var}ONE argv element (spaces do not split)blocked
${*var}shlex.split() → extend argv (no rejoin)inert
| in templatePipeline separator → Popen chainby design
| from userPlain token in argvblocked

Runtime container isolation

Default hardening (Docker)
read_only=True
tmpfs={"/tmp": "rw,noexec,nosuid,size=64m"}
mem_limit="512m"
nano_cpus=1_000_000_000        # 1 vCPU
cap_drop=["ALL"]
security_opt=["no-new-privileges:true"]
user="1000:1000"
OpenShift SCC On OCP do not set runAsUser: 1000 in the pod spec — OpenShift assigns a UID from the namespace range (e.g. 1000780000+). The manifests in MCCPLATFORMforKUBERNETES/k8s/ use the correct config (runAsNonRoot: true without a specific UID).

Security: Tool Policy

Each runtime has a policy.json file controlling what is allowed to execute. Set the policy in UI → server → Security (Policy tab).

Example policy.json — OpenShift read-only
{
  "allowed_binaries": ["oc", "kubectl", "jq"],
  "blocked_commands": ["delete", "apply", "patch", "exec"],
  "require_read_only": true,
  "block_write_tools": true,
  "block_destructive_tools": true,
  "timeout_seconds": 30,
  "max_response_bytes": 5242880,
  "require_approval_for": ""
}
Example policy.json — OpenShift with approvals
{
  "allowed_binaries": ["oc", "kubectl", "jq", "printf"],
  "blocked_commands": [],
  "require_read_only": false,
  "block_write_tools": false,
  "require_approval_for": "auto",
  "require_approval_for_prefixes": ["oc delete", "oc apply", "kubectl delete"],
  "approval_timeout_seconds": 300
}

Allowed binaries allowlist

The allowed_binaries field is checked for every pipeline stage independently. If the list is set, a binary not on the list is blocked even if the first stage passed validation.

Security: SSRF and Network

SSRF protection

The control plane validates every external URL before sending an HTTP request. The _is_safe_fetch_url() function:

  • Blocks schemes other than http/https
  • Blocks known local names: localhost, 169.254.169.254, metadata.google.internal
  • Blocks private IP addresses (RFC 1918, loopback, link-local)
  • For domain names: resolves DNS via socket.getaddrinfo() and checks every returned IP — protects against DNS rebinding
Applies to Tool package imports from URL, external MCP server registration, MCP endpoint probing.

Runtime network isolation

Runtime containers run in the Docker ai-net network (isolated from the host). The control plane and operator communicate over this network. External ports (19000–19800) are optionally exposed via MCP_RUNTIME_PUBLIC_PORT_BASE.

Cookies and sessions

User session stored in the mcp_session cookie:

  • httponly=True — not accessible from JavaScript
  • samesite="lax" — CSRF protection for cross-site navigation
  • secure=True — enable via MCP_HTTPS_ONLY=1 when platform is behind HTTPS termination (e.g. OpenShift Route)

Human-in-the-Loop Approval System NEW

Tools with write or destructive mode can require explicit user confirmation before execution. Approval happens directly in the AI chat — no separate web panel or extra UI actions needed.

How it works

AI calls tool
→
Runtime checks policy
→
⚠️ approval_required
→
AI asks in chat
→
User: "yes"
→
✅ AI calls again → execute

The runtime returns a message to the AI with the command description and instructions. The AI asks the user in chat. After confirmation, the AI calls the same tool again — the runtime automatically executes the command.

Two confirmation methods

MethodHow it worksCompatibility
Auto-confirm on repeated call The first call registers a pending approval. The second identical call (same tool + same args) within the approval_timeout_seconds window is automatically confirmed — no extra parameters needed. All AI models
__confirm parameter AI adds __confirm="yes" (or "true", "1", "ok", etc.) to the call parameters. The parameter is defined in every tool's input schema. Models that can add a new parameter to an existing call
Why auto-confirm works with every model Some models (e.g. local models like Qwen, Llama) cannot add the __confirm parameter to a repeated tool call. Auto-confirm solves this — the AI simply calls the same tool with the same arguments, which is the natural behaviour after user confirmation.

Configuration

require_approval_for valueBehaviour
not set or ""No approvals — default
"auto"Auto-detect: mode=write/destructive OR tool name contains action keyword (delete, create, apply, patch, scale…)
["destructive"]Only mode=destructive tools (delete)
["write", "destructive"]Both write (create/apply) and destructive (delete) tools
Recommended configuration — auto with command prefixes
{
  "require_approval_for": "auto",
  "require_approval_for_prefixes": ["oc delete", "oc apply", "kubectl delete"],
  "approval_timeout_seconds": 300
}
Do not mix with blocked_command_prefixes If you want approval for a prefix — use require_approval_for_prefixes. Do not add the same prefixes to blocked_command_prefixes. After user confirmation (__confirm="yes" or auto-confirm), the runtime skips the blocked prefix check — but allowed_binaries is still enforced.

Auto-detection keywords (mode "auto")

Approval required when tool name contains:

deleteremovedestroydroppurgewipetruncateeraseclean createapplydeployinstallpatchscaleexposerolloutaddsetupdatereplacerestart

MCP Authentication NEW

Each MCP server can be secured with a Bearer token that must be provided by the AI client (Claude Desktop, Cline, Continue, etc.). Without a token the server remains open — this feature is optional and per-runtime.

How it works

AI client
→
Authorization: Bearer <token>
→
Runtime middleware
→
✅ Authorized → tool

Missing or wrong token → 401 Unauthorized with WWW-Authenticate: Bearer header.

Enabling a token (UI)

  1. Go to the runtime detail page (Runtimes → name)
  2. Switch to the 🔐 Auth tab
  3. Click + Generate token
  4. Copy the token and paste it into your AI client configuration

Token is generated via secrets.token_urlsafe(32) — 256 bits of entropy. Rotation: click ↻ Regenerate token — the old token stops working immediately.

Zero downtime Changing the token does not require a container restart. The token is written to runtime-config.json and loaded via /reload automatically after generation.

Supported header formats

HeaderFormatDescription
AuthorizationBearer <token>Standard OAuth2 / RFC 6750 — recommended
X-API-Key<token>Simpler format — token only, no Bearer prefix
Environment variables do NOT replace the header Some clients (e.g. OpenCode) have an "Environment variables" section in the MCP configuration. For remote servers (transport remote-SSE or streamable-HTTP), those variables are not sent to the server — they only apply to local processes (stdio). The token must always go in a request header.

Client configuration

Claude Desktop (~/.claude.json)
{
  "mcpServers": {
    "my-server": {
      "type": "http",
      "url": "http://localhost:19001/mcp",
      "headers": {
        "Authorization": "Bearer <YOUR_TOKEN>"
      }
    }
  }
}
OpenCode — Request Headers (simpler variant with X-API-Key)
Transport:   remote - SSE
Server URL:  https://<runtime-route>/sse

Request Headers:
  Key:   X-API-Key
  Value: <YOUR_TOKEN>

# Do NOT use the "Environment variables" section — they don't reach a remote server.
# Leave OAuth unchecked.
Cline / Continue (settings.json)
{
  "mcp.servers": [{
    "name": "my-server",
    "transport": "streamable-http",
    "url": "http://localhost:19001/mcp",
    "headers": {
      "Authorization": "Bearer <YOUR_TOKEN>"
    }
  }]
}
curl — connection test
curl -H "Authorization: Bearer <YOUR_TOKEN>" \
     http://localhost:19001/health

# or with X-API-Key:
curl -H "X-API-Key: <YOUR_TOKEN>" \
     http://localhost:19001/health

Paths excluded from auth

The following endpoints are always public (no token required):

  • /health — container health check by operator
  • /reload — config reload by operator (runtime-shell and http-gateway only)

Token comparison security

Comparison uses secrets.compare_digest() — resistant to timing attacks. The token is never logged in any file or database other than runtimes.mcp_auth_token (plaintext — do not use in production without database encryption).

Warning: plaintext in SQLite The runtimes.mcp_auth_token column stores the token in plain text. In production environments, database encryption or an external vault is recommended. This is a known limitation documented in the architecture notes.

Runtime Lifecycle — Deploy, Configure, Reload

Every MCP server goes through several stages. Understanding what requires a Reload vs a Redeploy avoids unnecessary restarts.

First deployment

  1. Create runtime — UI → Quick Start (or Advanced Creator) → select class (e.g. shell-readwrite) → Create server.
  2. Add tools — Tools tab → + Add tool → fill in name, command, mode, argument schema.
  3. Set policy — Policy tab → set allowed_binaries, require_approval_for, etc. → Save.
  4. Add secrets / env vars — Secrets tab → + Add (values injected as env vars into the container).
  5. Deploy — click ▶ Deploy (or the operator auto-detects the new record). Status will change to running.
  6. Connect AI client — copy the MCP endpoint from the page header (e.g. http://localhost:19001/mcp) into Claude Desktop / Continue / Open WebUI config.

What only needs a Reload (no container restart)

Reload (POST /reload on the runtime) reloads config files from disk without killing the process. It is triggered automatically by the control plane after saving changes.

ChangeHow to applyAuto reload?
Edit tool (name, command, schema, mode)UI → Tools → Edit → Save✅ Yes — control plane calls /reload
Change policy (allowed_binaries, approval, timeouts)UI → Policy → Save shell policy✅ Yes
Change auth tokenUI → Auth → Generate / Regenerate token✅ Yes
Import tool packageUI → Import JSON✅ Yes
Reload does not interrupt running calls The runtime process keeps running — only config files are reloaded. MCP clients do not lose their connection.

What requires a Redeploy (container restart)

ChangeHow to apply
Adding / removing secrets (env vars)UI → Secrets → save → click 🔄 Redeploy
Changing runtime class (different Docker image)UI → Redeploy
Memory / CPU limit changeUI → Redeploy
New version of runtime-shell after ./deploy.shUI → Rebuild Image + Deploy (on K8s: triggers rollout restart)
Secrets require Redeploy Environment variables are injected into the container at start time. Changing a secret value is not visible to a running container until a Redeploy is performed.

How Reload works technically

The control plane writes artifacts to ./data/configs/<runtime-id>/ (Docker) or a ConfigMap (K8s), then calls POST /reload on the runtime endpoint. The runtime reloads:

  • tools.json — tool definitions with commands
  • policy.json — security policy
  • runtime-config.json — metadata + auth token
  • adapter-config.json — adapter configuration

runtime-env.json (env vars from secrets) is not reloaded — requires a process restart.

Manual reload (if auto reload failed)

# Docker — directly on the container
curl -X POST http://localhost:19001/reload

# OpenShift — via the runtime route
curl -X POST https://mcp-runtime-<id>-mcp-platform.apps.<domain>/reload

Troubleshooting

General

ProblemSolution
Runtime pod in ImagePullBackOff stateCheck that MCP_RUNTIME_IMAGE_REGISTRY_PREFIX points to the internal svc (not the external route). Pods cannot use the external route to pull images.
403 Forbidden on Rebuild Image + Deploy on K8sThere is no BuildConfig on K8s — the action correctly skips the build and only does a redeploy. If 403, check operator ServiceAccount permissions.
Runtime in state failed: SCC forbidden runAsUserRemove runAsUser: 1000 and fsGroup: 1000 from pod securityContext. Keep only runAsNonRoot: true.
oc: command not found in deploy.shDownload oc to the directory: wget https://...tar.gz && tar xvf oc.tar. deploy.sh automatically adds $PWD to PATH.
LVMCluster stuck ProgressingAfter cluster restart: sudo rm -f /var/lock/vgmanager/vgmanager.lock on the node, then delete the vg-manager pod.
LVMCluster Failed: already exists in filesystem(a) VG name collides with device name — change name to something other than vdc. (b) Old PV signatures on disk — pvremove -ff /dev/vdc.
docker push TLS error to OCP registryUse podman push --tls-verify=false instead of docker push. Docker requires a restart after adding insecure-registries.
Approval not working — AI calls again but still blockedCheck that it is the exact same tool with the exact same arguments. Auto-confirm only works when the hash (tool_name + args) is identical. Restarting the pod clears all pending approvals.
Approval window too shortSet approval_timeout_seconds in policy JSON (default 300s). Pending approvals are cleared on pod restart.
Test tab returns 500 Internal Server ErrorThe tool requires a mandatory parameter (e.g. args). Fill in the Arguments JSON field in the test form before clicking Run Tool.
